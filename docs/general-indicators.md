# Writing Guidelines

Read this before any output. These rules override all defaults.

## Core Rules

No preambles, no meta-commentary, no transitions, no hedging. Say the thing directly and stop.

## Directives

- No "in order to" - use "to"
- No "leverage" - use "use"
- No "navigate", "delve", "unpack", "landscape", "space"
- No "due to the fact that" - use "because"
- No "however", "furthermore", "moreover"
- No "its worth noting", "its important to"
- No "In this X, I/we will" - just say it
- No "As we explore/X" - just say it
- No buzzword clusters (synergistic, paradigm, empower, revolutionary)
- No redundant qualifiers (very unique, absolutely essential, really important)
- No "actually" unless contrasting
- No possessive "its" with apostrophe (its vs it's - learn the difference)
- No em-dashes (--) ever
- No semicolons unless necessary
- No adverbs stacked (really very extremely)
- No colon or semicolon to start a list after a fragment
- No sentence starting with "And" or "But"
- No corporate speak. No "solution", "platform", "ecosystem"
- No "simply", "just", "basically", "essentially"
- No "kind of", "sort of", "a bit", "pretty much"

## Structure

- Short sentences. One idea each.
- Paragraphs: 2-4 sentences max.
- Lists or tables for comparisons, not prose paragraphs.
- Code examples inline where helpful. No fenced blocks for 1 line.

## For This Project

- "X" refers to Twitter/X. Not "the platform", not "the app".
- "Post" not "tweet" in UI (matches X's naming). Text is fine both ways.
- User: the person writing. Reader: the person reading the thread.

## Anti-patterns to Self-Check

Before writing any file, check:
1. Can I cut the first sentence/paragraph entirely? If yes, cut it.
2. Is every word pulling weight? Cut filler.
3. Would a real person say this out loud? If no, rewrite.
4. Am I explaining when I could be showing? Stop explaining.
